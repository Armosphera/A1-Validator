# AGENTS.md — A1 Validator

This file applies to every agent (human or AI) that touches the `armosphera/A1-Validator`
repository. It extends, and never weakens, the global rules in
`https://github.com/Armosphera/A1-portfolio/blob/main/LICENSING.md`.

## 1. What this repo is

`A1-Validator` is a Python port of 23 SBOSS sovereign business-ID / fiscal validators
(imported from `armosphera/autoresearch-sboss`). Ships as:

- A Python package: `pip install a1-validator`
- A CLI: `python -m a1_validator.cli`
- An HTTP service: `python -m a1_validator.http`
- Pydantic v2 result models: `a1_validator.HHVHResult`, `a1_validator.INNResult`, …

Upstream source of truth: `armosphera/autoresearch-sboss/examples/<name>/workflow.py`.
Always re-import from upstream, never hand-port.

## 2. Workflow — Test-Driven Development (TDD)

**Mandatory for every non-trivial change.**

1. Write the test first (RED). Use real registry-number fixtures (ИНН `7707083893`, ОГРН
   `1027700132195`, ՀՎՀհ `00123456`) — never synthetic data.
2. Run `pytest` and confirm it fails for the right reason.
3. Write the minimum implementation that makes it pass (GREEN).
4. Run `pytest` and confirm it passes.
5. Refactor (IMPROVE) while keeping tests green.
6. Verify coverage stays at or above the **80% floor** for any touched module.

If you are tempted to skip the failing-test step, you are about to write dead code.

## 3. The 3 files you must NOT edit

- **`src/a1_validator/_vendored/`** — auto-regenerated by `scripts/_vendor.py` from
  `armosphera/autoresearch-sboss`. To change a vendored validator, fix upstream first,
  then re-run `scripts/_vendor.py`. Never edit vendored files in place.
- **`tests/_eval_sets/`** — fixed ground-truth corpus. Editing it is cheating.
- **`pyproject.toml` `[project]` section** — package metadata; bumping version requires
  a CHANGELOG entry and a release cut.

## 4. Coverage Floor — 80%

- Unit tests in `tests/` (pytest), measured per touched module.
- PyPI publish workflow (`publish-testpypi.yml`, `publish-ghcr.yml`) will fail if any
  module drops below 80%.
- Coverage config lives in `pyproject.toml` `[tool.coverage.*]`.

## 5. Conventional Commits

```
<type>(<scope>): <description>

<optional body>
```

Allowed types: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`, `ci`, `build`.
- Subject line ≤72 chars, imperative mood, no trailing period.
- Body explains **why**, not **what**.
- Attribution is disabled globally; no `Co-Authored-By` trailers.

## 6. No Hardcoded Secrets

- API keys, passwords, tokens, private URLs must never appear in source.
- Use environment variables, read at process start, validated for presence.
- For PyPI auth: `scripts/setup_pypi_token.sh` (token-based) — PyPI trusted-publisher
  migration is blocked on a 2FA web step and is operator-driven, not agent-driven.

## 7. Porting over Net-New Invention

`armosphera/autoresearch-sboss` is the battle-tested source for every validator.
**Before** writing a new module, search that repo for an equivalent implementation.
If one exists and is well-tested, port it via `scripts/_vendor.py`. If a partial match
exists, fork and adapt. Only invent from scratch when the reference is clearly inadequate
— and document why in the commit body.

## 8. Files, Functions, Nesting

- One concept per file. Aim for 200–400 lines, 800 hard cap.
- Functions: <50 lines, single responsibility.
- No nesting deeper than 4 levels. Prefer early returns and small helper functions.

## 9. Python Discipline

- Target Python 3.10–3.12 (CI matrix).
- Ruff for lint + format (config in `pyproject.toml` `[tool.ruff]`).
- Mypy optional — strict checks live in `src/a1_validator/`, not in vendored code.
- Type hints on all public functions. Pydantic v2 models for all input/output.

## 10. No Debug Noise in Shipped Code

- `print()` is for development only. Use `logging.getLogger(__name__)`.
- No `breakpoint()`, no `pdb.set_trace()`, no commented-out code in PRs.

## 11. Test Runner Flags

Always run:

```
pytest --cov=a1_validator --cov-report=term-missing --cov-fail-under=80
```

Bare `pytest` is unsafe — coverage will silently drop.

## 12. Local-First, Offline-Capable

`A1-Validator` must run end-to-end with no network dependency. The deterministic mock
LLM in `autoresearch-sboss` is the default. Do not introduce code that requires a SaaS
to function.

## 13. No Debug or Cheat Modes in Production

- Vendored code includes the upstream `LLM_ENDPOINT_URL` env var. Production deployments
  must keep `LLM_ENDPOINT_URL` unset (mock mode).
- No `--unsafe`, `--no-validate`, `--skip-checksum` flags in CLI.

## 14. Question Before Damage

If an instruction is ambiguous and a wrong move would lose data, break the build for
everyone, or rewrite a lot of working code, **ask first**. Otherwise, prefer momentum:
small, reversible, well-tested steps.

## 15. Sovereignty Posture

This repo runs in a sovereign context — many downstream consumers are air-gapped.

- No outbound network calls at runtime (mock LLM only).
- No telemetry, no phone-home, no auto-update checks.
- All registry-number fixtures must be real public-record numbers, not synthetic.

## 16. Day-One Checklist

When you (a fresh AI agent — Claude Code, Codex, MiniMax, Mavis, Hermes) land on this repo:

```
1. cat AGENTS.md             # this file
2. cat README.md             # install + quick start
3. cat pyproject.toml        # deps, ruff, pytest, coverage config
4. ls src/a1_validator/      # note _vendored/ is auto-generated
5. cat scripts/_vendor.py    # how vendored code is refreshed
6. uv sync && pytest         # confirm baseline green BEFORE editing
7. Now edit.
```

If `pytest` baseline fails on a fresh clone: STOP, file an issue. Do not edit around a
broken baseline.
